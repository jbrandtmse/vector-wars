# Story 5.10: Ending Sequence

Status: review

## Story

As a player,
I want to experience the ending sequence with outro music and credits so that completing the game feels rewarding and earned.

## Acceptance Criteria

1. **EndingScreen class** exists at `src/ui/screens/EndingScreen.ts`. It is an HTML overlay (same pattern as BriefingScreen/GameOverScreen) that manages the full ending sequence: network collapse text, Ghost's final transmission lines, white flash, credits scroll, and restart prompt. It is NOT skippable (per narrative design -- the ending is the earned reward).

2. **EndingScreen displays Ghost's final transmission** as sequential timed text lines overlaid on a dark background:
   - Line 1: "CIPHER! THE NETWORK'S COLLAPSING--" (desperate tone, appears immediately after white flash)
   - Line 2: "I'M LOSING YOUR SIGNAL--" (1.5s delay)
   - Line 3: "STAY WITH ME! STAY WITH ME!" (1.5s delay)
   - Line 4: (pause 2s, static-like text flicker)
   - Line 5: "...I've got you. You're coming home." (quiet, relieved -- different styling: lowercase, softer glow)
   Lines use palette-aware colors via `getPaletteHexColor()` / `getPaletteCSSGlow()`. The final line uses reduced opacity/glow to convey the shift from desperate to quiet relief.

3. **Credits scroll** begins after Ghost's transmission completes (~8s after start). Credits display in vector-styled text (Courier New, palette glow) scrolling upward. Content includes: game title "VECTOR WARS", "A game by [Developer]", sections for Design, Code, Audio, Narrative, Tools (Three.js, Vite, Web Audio API), and "Thank you for playing". Credits scroll at the same speed as briefing screens (`BRIEFING_SCROLL_SPEED`).

4. **Outro music plays** during the ending sequence. A procedural "outro" audio buffer is generated by a new `OutroMusicGenerator` class at `src/audio/OutroMusicGenerator.ts`. This generates a ~60-second ambient/melodic audio buffer using Web Audio API (OfflineAudioContext) -- layered sine/triangle oscillators with slow chord progression and reverb, creating an atmospheric, hopeful tone. The generator follows the same pattern as `SFXGenerator`/`VoiceLineGenerator` (generate method with cache).

5. **EndingScreen plays outro music** via `audioManager.playMusic('outro')` on enter. AudioManager already supports the `playMusic` method. The OutroMusicGenerator is registered with AudioManager as a music generator. AudioManager's `playOnChannel` method must be extended to check the music generator if no manifest entry or file buffer is found (same pattern used for SFX/voice generators).

6. **Ambient hum stops** when the ending sequence starts. EndingScreen calls `audioManager.stopChannel('ambient')` before playing outro music.

7. **EndingScreen shows restart prompt** after credits complete: "PRESS SPACE TO PLAY AGAIN" with pulsing animation. Space key triggers `window.location.reload()` (same pattern as GameOverScreen).

8. **main.ts levelComplete handler updated**: When level 3 completes, instead of the current placeholder "MISSION COMPLETE" overlay, it creates and shows the EndingScreen. The placeholder victory overlay code (lines ~277-297) is replaced with EndingScreen instantiation.

9. **Ghost ending voice lines** added to VoiceLineGenerator: `ending_desperate` (L3 profile, 3.0s duration) for the desperate lines, and `ending_relief` (L1 profile, 2.5s duration) for the quiet relief line. EndingScreen plays these via `audioManager.playVoice()` timed to match text display.

10. **White flash effect** on ending entry: A brief white overlay (`opacity: 0 -> 1 -> 0` over ~1s) simulates the "jack out" moment before Ghost's transmission begins. Implemented as a CSS-animated div within EndingScreen.

11. Running `npx tsc --noEmit` produces zero TypeScript errors.

12. Unit tests exist (Vitest) for:
    - EndingScreen: creates overlay DOM element on show()
    - EndingScreen: displays Ghost transmission lines in sequence with correct timing
    - EndingScreen: shows credits section after transmission completes
    - EndingScreen: shows restart prompt after credits
    - EndingScreen: calls audioManager.playMusic('outro') on show
    - EndingScreen: calls audioManager.stopChannel('ambient') before playing music
    - EndingScreen: calls audioManager.playVoice() for ending voice lines
    - EndingScreen: Space key triggers page reload after restart prompt appears
    - EndingScreen: is NOT skippable during transmission/credits phases
    - OutroMusicGenerator: generate() returns an AudioBuffer
    - OutroMusicGenerator: generates buffer with correct duration (~60s)
    - OutroMusicGenerator: caches buffer on subsequent calls
    - VoiceLineGenerator: ending voice line IDs exist (ending_desperate, ending_relief)
    - VoiceLineGenerator: total voice line count updated (59 -> 61)

13. All existing tests continue to pass -- zero regressions.

## Dev Notes

### Architecture Compliance

- **Systems never import each other.** EndingScreen is a UI screen component (same layer as BriefingScreen, GameOverScreen). It imports `audioManager` singleton directly (same pattern as BriefingPhase). [Source: project-context.md#Architecture Rules]
- **UI never imports game logic.** EndingScreen receives finalScore as a parameter. It does not reach into ScoreManager or any game system. [Source: project-context.md#Architecture Rules]
- **Use `Logger.info/warn/error()`** -- NEVER `console.log()`. [Source: project-context.md#Critical Don't-Miss Rules]
- **`erasableSyntaxOnly: true`**: No `enum`. Use `type` unions and `as const` objects. [Source: tsconfig.json]
- **`verbatimModuleSyntax: true`**: Use `import type { ... }` for type-only imports. [Source: tsconfig.json]
- **No per-frame allocations during gameplay.** The ending sequence runs after gameplay ends, so per-frame concerns are relaxed. However, still avoid wasteful allocations in animation loops. [Source: project-context.md#Performance Rules]
- **No `fetch()` during gameplay frames.** OutroMusicGenerator generates procedurally -- no network requests. All generation happens before playback starts. [Source: project-context.md#Performance Rules]

### Critical Implementation Rules

- **VoiceLineGenerator uses `createVoiceDefinition()` factory.** All voice definitions MUST be created with this function. Follow existing pattern exactly.
- **Per-level handler profiles already exist.** `HANDLER_PROFILE_L1` (calm, 180Hz), `HANDLER_PROFILE_L3` (desperate, 220Hz). Use L3 for desperate lines, L1 for relief line.
- **audioManager.playVoice(id)** plays a voice line by ID. The VoiceLineGenerator generates the audio buffer on first call, then caches it.
- **audioManager.playMusic(id)** plays on the music channel. Currently only checks manifest/file buffers. Must be extended to check a music generator (OutroMusicGenerator).
- **audioManager.stopChannel(channelType)** stops playback on a channel. Already implemented -- use to stop ambient hum.
- **BriefingScreen pattern**: EndingScreen should follow the same DOM creation and lifecycle pattern (constructor creates elements, show() displays, dispose() cleans up).
- **GameOverScreen pattern**: Restart via `window.location.reload()` after space key press, with delay guard.
- **Ending is NOT skippable** per narrative design. No skip prompt, no key handler for skip during transmission/credits. Only the restart prompt after credits complete responds to input.
- **Credits are simple.** This is a solo developer project. Keep credits brief: game title, developer credit, tools used, "Thank you for playing".
